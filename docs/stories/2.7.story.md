# Story 2.7 — Clarification Pré-Action : Discussion Fluide Avant Exécution

## User Story

**En tant qu'** utilisateur d'Ami,
**Je veux** qu'Ami me pose les questions nécessaires AVANT de naviguer sur le web,
**Afin que** la navigation soit faite du premier coup avec toutes les informations correctes, sans interruptions ni allers-retours une fois l'action lancée.

---

## Contexte & Motivation

**Avant (comportement v1) :**
```
Utilisateur : "Je veux aller chez le dentiste"
Ami → lance Doctolib → "Je cherche..." → résultats → "C'est pour quelle ville ?"
```
Problème : on lance une navigation générale, on attend 5-15s, et seulement après on pose des questions. Latence inutile + mauvaise UX.

**Après (comportement v2) :**
```
Utilisateur : "Je veux aller chez le dentiste"
Ami → "C'est pour quelle ville ?" (TTS rapide, <1s)
Utilisateur : "Lyon"
Ami → "Plutôt quand ?" (si nécessaire, sinon passe directement à l'action)
Utilisateur : "Demain matin"
Ami → "Je cherche sur Doctolib..." → lance OpenClaw("dentiste Lyon demain matin") → résultat précis
```
Avantage : discussion fluide à faible latence, puis une seule navigation avec les bons paramètres.

---

## Critères d'Acceptation

1. **Étant donné** une demande vague (ex : "je veux un dentiste"), **quand** le LLM détecte qu'une info critique manque, **alors** Ami pose UNE question (via TTS) avant de naviguer.
2. **Étant donné** qu'Ami pose une question, **quand** l'utilisateur répond, **alors** Ami utilise cette réponse dans sa tâche OpenClaw (enrichissement de la task string).
3. **Étant donné** 2 questions déjà posées, **quand** il manque encore des infos, **alors** Ami navigue quand même avec ce qu'elle a (pas de 3ème question).
4. **Étant donné** une demande complète (ex : "dentiste à Paris demain"), **quand** toutes les infos sont présentes, **alors** Ami ne pose AUCUNE question et navigue directement.
5. **Étant donné** qu'une question est posée, **quand** l'utilisateur répond, **alors** la latence entre la réponse et le lancement d'OpenClaw est < 1s.
6. **Étant donné** que l'utilisateur interrompt pendant la question de clarification, **alors** le contexte est réinitialisé correctement.

---

## Nouveaux États Machine d'État

```
CLARIFYING → état léger (pas d'OpenClaw, pas de navigation)
           → LLM vient d'émettre need_info
           → TTS joue la question
           → Revient en LISTENING pour attendre la réponse
           → Sur transcription : reprend processTranscript(transcript, isClarificationReply=true)
```

## Notes Dev

- **Pourquoi `need_info` comme function call et non du texte libre ?** — Le function call force une structure JSON (`{ question }`) que la session peut parser et router sans ambiguïté. Un texte libre ne permettrait pas de distinguer "question" vs "narration".
- **`isClarificationReply` flag** : stocké sur l'instance Session entre deux turns (à cheval sur deux `onTranscript`). Reset à `false` dès lecture, reset complet dans `resetHistory()`.
- **`clarifyCount` max 2** : contrainte imposée PAR le system prompt (le LLM ne peut appeler `need_info` que 2 fois), mais aussi défensivement par le compteur dans la Session (en cas d'hallucination LLM).
- **Latence** : une question `need_info` = un appel LLM + TTS. Latence ~800ms–1.2s pour entendre la question. Bien inférieure aux 5-15s d'une navigation OpenClaw inutile.
- **Compatibilité ascendante** : `processTranscript(transcript)` continue de fonctionner sans le 2ème paramètre — valeur par défaut `false`.

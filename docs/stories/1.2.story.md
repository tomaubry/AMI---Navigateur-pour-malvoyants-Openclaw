# Story 1.2 — Installation et Configuration d'OpenClaw sur VPS

## User Story

**En tant que** développeur,  
**Je veux** installer et faire tourner le daemon OpenClaw comme service système persistant sur le VPS,  
**Afin que** le MCP Server puisse lui envoyer des tâches de navigation à tout moment sans intervention manuelle.

---

## Critères d'Acceptation

1. **Étant donné** que Node.js et Chromium sont installés (Story 1.1), **quand** j'exécute le script d'installation officiel depuis `openclaw.ai`, **alors** l'installation se termine sans erreur.
2. **Étant donné** qu'OpenClaw est installé, **quand** j'exécute `systemctl status openclaw`, **alors** le statut du service affiche `active (running)`.
3. **Étant donné** qu'OpenClaw tourne, **quand** j'exécute le script de test local `node execution/test_openclaw.js "Ouvre Wikipedia et lis le premier paragraphe"`, **alors** le daemon accepte la commande et retourne une réponse textuelle (contenu extrait du DOM).
4. **Étant donné** que le VPS redémarre, **quand** le serveur revient en ligne, **alors** le service OpenClaw redémarre automatiquement (`systemctl is-enabled openclaw` → `enabled`).
5. **Étant donné** qu'OpenClaw tourne, **quand** je consulte les logs du service via `journalctl -u openclaw -n 50`, **alors** aucune erreur critique n'est présente.
6. **Étant donné** que le gateway est configuré, **quand** j'exécute `openclaw security audit`, **alors** aucun finding de sévérité `critical` n'est présent.

---

## Notes Dev

- **⚠️ Sécurité critique — Anti-typosquatting :** Ne JAMAIS installer OpenClaw via un package NPM non officiel (ex: `@openclaw-ai/openclawai`). Toujours utiliser le script officiel de `openclaw.ai`
- **⚠️ Sécurité critique — Pas de root :** La doc officielle interdit explicitement de faire tourner OpenClaw en root. User dédié `openclaw` obligatoire
- **⚠️ Sécurité critique — Gateway auth :** Sans `gateway.auth.token`, tout process local peut appeler le gateway. Token fort (32 bytes hex) obligatoire
- **⚠️ Sécurité critique — Loopback only :** Le gateway doit être lié à `127.0.0.1` uniquement (`bind: "loopback"`). Ne jamais exposer sur `0.0.0.0`
- **curl TLS forcé :** L'option `--proto '=https' --tlsv1.2` bloque les downgrade HTTP et les certificats faibles — recommandée par la doc officielle
- **`openclaw security audit` :** Commande officielle à lancer après toute installation ou modification de config. Supporte `--fix` pour corrections automatiques
- **Contrat d'API :** OpenClaw expose une API HTTP compatible OpenAI (`/api/v1/chat/completions`). Le test script devra utiliser ce format (pas WebSocket)
- **Chromium snap incompatible :** Sur Ubuntu 24.04, Chromium est un snap package. Snap ne fonctionne pas correctement dans un service systemd. Solution : installer Playwright et utiliser son Chromium bundlé (`npx playwright install chromium`). Configurer `browser.executablePath` dans openclaw.json.
- **Xvfb requis :** Le browser tool OpenClaw nécessite un display X11. Installer xvfb, créer le service `xvfb.service`, et ajouter `DISPLAY=:99` dans `/etc/systemd/system/openclaw.service.d/display.conf`.
- **Device pairing :** Après le premier démarrage du gateway avec auth token, le CLI doit être "paired". Utiliser `openclaw devices list` puis `openclaw devices approve <id>` (avec `OPENCLAW_GATEWAY_TOKEN` en env var).
- **tools.profile = "full" :** Pour activer le browser tool dans les requêtes de l'agent. Les profils valides : `minimal`, `coding`, `messaging`, `full`.
- **Performance :** Tester avec une requête simple d'abord, puis une requête plus complexe (scroll, extraction de tableau) pour valider la capacité du VPS

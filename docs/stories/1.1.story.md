# Story 1.1 — Setup VPS Hetzner (Environnement de base)

## User Story

**En tant que** développeur,  
**Je veux** provisionner un VPS Hetzner avec toutes les dépendances système installées (Node.js, Chromium, Nginx),  
**Afin d'** avoir un serveur stable avec accès root, prêt à héberger le daemon OpenClaw et le backend MCP Server.

---

## Critères d'Acceptation

1. **Étant donné** que j'ai créé un compte Hetzner Cloud, **quand** je provisionne une instance CX32 avec Ubuntu 22.04 LTS, **alors** je peux me connecter en SSH avec l'utilisateur root.
2. **Étant donné** que je suis connecté en SSH, **quand** j'exécute `node --version`, **alors** la sortie affiche `v22.x.x` ou supérieur.
3. **Étant donné** que Node.js est installé, **quand** j'exécute `chromium-browser --version` (ou `chromium --version`), **alors** Chromium est présent et la commande s'exécute sans erreur.
4. **Étant donné** que le serveur tourne, **quand** j'exécute `nginx -t`, **alors** Nginx est installé et la syntaxe de configuration est valide.
5. **Étant donné** que Nginx tourne, **quand** j'accède à `http://<IP_SERVEUR>/health`, **alors** Nginx sert une réponse de base (page placeholder ou 200 OK).
6. **Étant donné** que le serveur est configuré, **quand** je vérifie `ufw status`, **alors** seuls les ports 22, 80 et 443 sont ouverts sur internet.

## Notes Dev

- **Taille d'instance :** CX32 minimum (4 vCPU, 8 Go RAM) — Chromium headless consomme ~1-2 Go RAM par session
- **NVM vs NodeSource :** NodeSource est recommandé pour un usage serveur (systemwide), NVM est mieux pour le dev local
- **Chromium vs Chrome :** Utiliser `chromium-browser` du dépôt Ubuntu pour éviter les dépendances propriétaires
- **Chromium root + no-sandbox :** Sur ce VPS (root), Chromium requiert `--no-sandbox` pour s'exécuter. À passer dans la config OpenClaw/Puppeteer en stories 2.x. Version installée : `146.0.7680.80 snap`
- **PM2 :** Sera utilisé pour gérer le process MCP Server en section 2.x. L'installer ici pour valider dès le départ
- **UFW :** Le port 3000 du MCP Server ne doit PAS être exposé directement — il passe par Nginx (proxy interne)

## En clair — Ce qu'on a fait

On a loué un ordinateur dans un datacenter en Finlande (le VPS Hetzner). C'est le serveur qui va faire tourner l'assistant vocal 24h/24, même quand ton propre ordinateur est éteint.

Sur ce serveur, on a installé tout ce dont le projet a besoin :

- **Node.js** — le moteur qui va exécuter le code de l'assistant
- **Chromium** — un navigateur web "invisible" (sans écran) que l'assistant utilisera pour naviguer sur internet à ta place
- **PM2** — un gardien qui surveille l'assistant et le redémarre automatiquement s'il plante
- **Nginx** — un portier qui reçoit les appels entrants (depuis WhatsApp par exemple) et les redirige vers l'assistant
- **UFW** — un pare-feu qui bloque tout accès non autorisé au serveur, sauf les portes nécessaires (SSH pour nous y connecter, HTTP/HTTPS pour les appels entrants)

En résumé : le serveur est propre, sécurisé, et équipé. Il attend maintenant qu'on y dépose le code de l'assistant.
